# 统一微信入口账号与动态 Agent 绑定方案

> **Status**: Draft for implementation | **Last Updated**: 2026-04-22 | **Purpose**: 固定“单一微信入口账号服务多用户”的主方案，并预留“统一二维码后为用户创建专属逻辑 agent”的备用方案，同时明确它与当前会话槽位释放机制的边界

## 1. 目标

本方案用于回答三个已经明确的产品目标：

- 使用一个固定的微信入口账号服务很多外部用户
- 外部用户共享一个稳定入口，而不是反复扫描新的登录二维码
- 每个用户仍然拥有独立会话、独立上下文，并可被分发到不同节点处理

同时，本方案预留一个后续增强方向：

- 用户通过统一入口接入后，系统可为该用户自动创建一个专属逻辑 agent 绑定

当前这份文档不改变现有后端协议事实，只固定正确的产品语义和实现边界。

## 2. 当前协议边界

当前微信接入基于 iLink / openclaw-weixin 这一类登录协议，必须把两类二维码彻底区分：

- **登录二维码 / onboarding 二维码**
  用于让管理员把某个微信入口账号登录到系统里，生成 `WECHAT_TOKEN`
- **公共入口二维码**
  用于让外部用户知道应该联系哪个固定微信入口账号

当前已确认的协议现实是：

- 新的登录二维码被另一人扫码确认后，会顶掉旧的登录态
- 因此登录二维码不能作为“很多终端用户共用的统一公共二维码”
- `openakita` 的多二维码能力，本质上是“多入口账号登录”，不是“一个二维码让很多用户同时绑定同一个 bot”

所以当前主方案必须建立在：

- 一个固定入口微信号
- 一个固定公共入口资料
- 网关内部按用户分 session

而不是继续把 onboarding 二维码当作用户入口。

## 3. 主方案：单一入口账号服务多用户

## 3.1 基本定义

主方案固定采用：

- 一个 `WECHAT_TOKEN` 对应一个固定微信入口账号
- 外部用户直接联系这个入口账号
- 网关从微信事件中读取 `from_user_id`
- 每个外部用户都映射成独立主会话

当前会话键继续保持：

```text
session_id = "wechat:{from_user_id}"
```

这意味着：

- 一个用户对应一个主会话
- 同一个入口账号下，不同用户天然隔离上下文
- 节点分发仍以“会话”为单位，而不是以“入口账号”为单位

## 3.2 公共入口二维码的定义

本方案中的“统一二维码”固定定义为：

- 固定入口微信号自己的联系人二维码

系统需要做的不是重新生成这张二维码，而是：

- 保存它的展示资料
- 在接入中心展示给运营人员
- 明确告诉用户这是“对外入口资料”，不是“系统登录二维码”

建议控制台后续补充一组公共入口资料：

- `public_entry_enabled`
- `public_entry_display_name`
- `public_entry_qr_url`
- `public_entry_notes`
- `public_entry_contact_hint`

这组资料只参与展示和运营说明，不参与微信协议登录。

## 3.3 与当前调度模型的关系

主方案与当前调度模型完全兼容：

- 会话仍由网关维护主状态
- 节点仍按 `assigned_node_id + assigned_slot_id` 处理当前会话
- `/switch` 和自动重分配仍围绕会话进行
- 通道释放仍然只释放“会话运行态槽位”，而不是用户身份或入口账号绑定

因此：

- 一个固定入口账号可以承载很多用户
- 每个用户仍可在不同时间被分发到不同节点
- 当前通道释放逻辑不需要因为“统一入口账号”而重写

## 4. 备用方案：接入后自动创建专属逻辑 Agent

## 4.1 适用目标

备用方案不是“多入口账号”，而是：

- 用户仍然通过同一个固定微信入口账号接入
- 系统在首次识别该用户时，为其分配一个专属逻辑 `agent_id`

这个 `agent_id` 的用途应固定为：

- 该用户的默认 persona / prompt 来源
- 该用户的默认业务配置来源
- 该用户的长期逻辑绑定

它**不**应直接等同于：

- 一个永久占用的节点进程
- 一个永久占用的节点通道
- 一条额外的主 session key

## 4.2 必须遵守的兼容边界

为了不和当前“通道释放”机制冲突，备用方案必须固定以下边界：

### 边界 1：扫码接入不立即占用槽位

用户接入后可以立即完成：

- 用户档案创建
- 逻辑 `agent_id` 绑定
- 欢迎消息或初始化提示

但不能立即完成：

- `assigned_slot_id` 获取
- `assigned_node_id` 长期绑定
- 持久化占用某个节点通道

节点槽位仍应在用户**真正发起消息**时，通过现有调度链路按需申请。

### 边界 2：通道释放只影响运行态，不影响 agent 绑定

当前运行态释放围绕：

- `assigned_node_id`
- `assigned_slot_id`
- `slot_expires_at`
- `active_task_id`

备用方案新增的“用户专属逻辑 agent”不能因为槽位释放而丢失，也不能反过来阻止槽位释放。

换句话说：

- **用户-agent 绑定** 是长期逻辑关系
- **session-slot 绑定** 是短期运行关系

这两层必须分开。

### 边界 3：本轮不引入“同一用户多个并行 agent 主会话”

当前主会话键仍固定为：

```text
wechat:{from_user_id}
```

因此备用方案当前只允许：

- 一个用户对应一个默认逻辑 `agent_id`

而不允许直接引入：

- 一个用户在同一微信入口下同时拥有多条长期独立主会话

如果未来要支持“同一用户切换多个 agent 且上下文完全隔离”，那将是新的 session key 设计专题，不能在本轮混入。

## 5. 推荐内部模型

为了同时兼容主方案和备用方案，推荐后续把内部模型明确拆成三层：

### 5.1 入口层

- 固定微信入口账号
- 固定公共入口二维码资料
- 微信协议登录态

### 5.2 逻辑绑定层

- `user_id -> default_agent_id`
- 用户专属 persona / prompt / 配置来源
- 用户长期档案

### 5.3 运行调度层

- `session_id -> assigned_node_id + assigned_slot_id`
- `queue_status`
- `active_task_id`
- `slot_expires_at`

这样分层后：

- 入口账号更换登录态不会影响用户逻辑绑定模型
- 槽位释放不会清空用户 agent 绑定
- 后续要做“首次接入自动创建 agent”时，只需要增补逻辑绑定层，不需要推翻当前 dispatch 和 release 机制

## 6. 建议实现顺序

## 6.1 第一阶段：先完成主方案的产品收口

先完成：

- 控制台明确区分“登录二维码”和“公共入口二维码”
- 新增公共入口资料展示
- 文档统一改成“固定入口账号服务多用户”的表述

这一阶段不引入新的会话键，也不引入新的节点占用语义。

## 6.2 第二阶段：补用户默认 agent 绑定

在主方案稳定后，再增加：

- 首次识别新用户时分配默认 `agent_id`
- 把该 `agent_id` 写入用户档案或 session 元数据
- 后续 dispatch 默认从这个逻辑绑定中取 agent 配置

这一阶段依然不做“扫码即占槽”。

## 6.3 第三阶段：再评估是否要支持更强的多 agent 模型

只有在确认确实需要“同一用户可拥有多套完全隔离 agent 上下文”时，才开启新的专题评估：

- session key 是否扩展为包含 `agent_id`
- 控制台如何展示同一用户下的多个线程
- `/switch` 和通道释放是否仍按主会话工作

这不属于本轮实现范围。

## 7. 测试与验收建议

主方案至少需要验证：

- 两个外部用户通过同一入口账号发消息时，生成两个不同的 `wechat:{user_id}` 会话
- 两个会话能被分配到不同节点或不同槽位
- 某个会话槽位被释放后，不影响另一个会话
- 更换登录 token 或重新扫码绑定入口账号后，公共入口资料仍不应被错误重置

备用方案预留至少需要验证：

- 某用户首次接入时可以被分配默认 `agent_id`
- 该逻辑绑定不会因为 `slot_expires_at` 到期而丢失
- 会话空闲释放后，用户下次发消息仍能沿用同一个默认 `agent_id`
- 当前 `dispatch_slot_released`、`dispatch_switch_timeout_release`、`/switch` 行为不因为逻辑 agent 绑定而失真

## 8. 当前结论

当前最推荐的落地方向是：

- **主方案**：单一微信入口账号 + 固定公共入口二维码资料 + 用户级独立会话 + 现有节点分发
- **备用方案**：用户首次接入后自动创建专属逻辑 `agent_id`，但不立即占用节点通道

这两者可以兼容推进。

真正不推荐的实现方式是：

- 继续把 onboarding 登录二维码当作公共入口二维码
- 把“每个用户一个专属 agent”实现成“每个用户永久占用一个节点通道”
- 在未重构 session key 的前提下直接引入“同一用户多个长期并行 agent 主会话”
