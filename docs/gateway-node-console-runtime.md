# Gateway / Node / Console 运行模型说明

## 1. 目标

本文档用于固定 `wechat-claw-hub` 当前推荐的运行模型，回答以下问题：

- 为什么网关、节点、控制台应保持中心化协作，而不是优先引入 P2P
- 当员工介入会话、代替 agent 与微信用户沟通时，系统边界应如何划分
- 在单网关前提下，如何控制网关压力，避免其成为单点性能瓶颈

本文档是对 [architecture-overview.md](/D:/wechat-claw-hub/docs/architecture-overview.md) 的补充，重点描述运行时职责、状态流向和后续优化路线。

## 2. 结论摘要

- 当前项目优先采用 `单网关 + 多节点 + 控制台` 的中心化模型，不优先引入 P2P
- 网关是唯一会话真源，也是唯一允许向微信出站消息的组件
- 节点只负责执行 AI 任务，不直接与员工协作，不直接向微信发消息
- 控制台只通过网关参与会话，不直接连节点，不直接连微信
- 员工参与会话后，更需要中心化状态机和统一审计，因此不应通过 P2P 分散会话控制权
- 网关压力的主要来源是“协调与状态管理”，不是“模型推理”；优化重点应是长连接、状态分层、异步化，而不是去中心化

## 3. 三端职责边界

### 3.1 网关 `Gateway`

网关负责：

- 微信接入与微信出站
- 会话创建、恢复、关闭
- 最近消息窗口与摘要状态维护
- 节点注册、鉴权、健康检查、调度
- 人工接管状态机
- 控制台实时事件推送
- 审计日志与诊断聚合

网关不负责：

- 大模型重推理
- 节点本地工具执行
- 员工界面的渲染与本地缓存

### 3.2 节点 `Node`

节点负责：

- 主动连接网关并完成注册
- 执行 AI 任务
- 调用 Dify / OpenAI 兼容接口
- 上报心跳、负载和错误
- 接收网关下发的 token、配置和任务

节点不负责：

- 保存长期会话真相
- 直接接入微信
- 直接与员工会话
- 裁决“AI 是否应继续发言”

### 3.3 控制台 `Console`

控制台负责：

- 展示会话列表和会话详情
- 展示网关、节点、微信的运行摘要
- 认领会话、发送人工消息、释放会话
- 查看节点诊断和会话运行轨迹

控制台不负责：

- 直接管理节点运行逻辑
- 直接向微信发送消息
- 直接从节点读取会话真相

## 4. 为什么当前不优先做 P2P

## 4.1 当前系统的核心问题不是链路长度，而是状态一致性

当前系统最核心的运行事实包括：

- 微信消息是否已经进入网关
- 当前会话是 AI 回复还是人工接管
- 当前节点是否在线、是否已注册、是否鉴权通过
- 这条回复最终是否允许发送给微信用户

这些判断都需要一个统一裁决者。即使引入 P2P，最终也仍需要回到网关确认，因此 P2P 不会消除核心复杂度。

## 4.2 P2P 会明显增加排障和运维复杂度

如果引入 P2P，需要额外处理：

- NAT 穿透与打洞
- 内网地址和公网地址漂移
- Windows 防火墙与安全软件拦截
- 节点之间的连接建立与重试
- 连接权限与端到端鉴权
- 失联节点的状态回收

对当前项目阶段而言，这些复杂度远高于收益。

## 4.3 员工参与会话后更需要中心化

一旦员工代替 agent 与微信用户沟通，系统必须保证：

- 员工与 AI 不会同时向微信发出冲突回复
- 节点正在运行的旧结果不会在人工接管后继续发出
- 每条发往微信的消息都能追溯“是谁发的”
- 会话认领、释放和切换都有统一审计

这些要求天然指向“统一网关中心”而不是 P2P。

## 5. 员工参与会话时的推荐模型

## 5.1 网关必须是唯一出站通道

无论消息来自：

- 微信用户
- AI 节点
- 员工控制台
- 系统事件

都必须经过网关，最终也只能由网关发往微信。

这样做的好处：

- 避免员工与 AI 重复回复
- 避免节点绕过人工接管规则直接发信
- 保留完整审计链
- 方便统一做敏感词、限流、权限和风控

## 5.2 推荐的会话状态机

建议会话至少具备以下状态：

- `bot_active`
- `handoff_pending`
- `human_active`
- `handoff_releasing`
- `closing`

建议同时维护：

- `claimed_by`
- `claimed_at`
- `handoff_ticket_id`
- `assigned_node_id`
- `active_task_id`

状态解释：

- `bot_active`：AI 正常工作，节点可继续执行
- `handoff_pending`：等待员工认领，AI 进入暂停或保守模式
- `human_active`：员工独占对话，AI 不得直接对外发信
- `handoff_releasing`：人工释放，准备恢复 AI
- `closing`：会话即将结束或已关闭

## 5.3 员工参与的两种模式

建议在产品上区分两类员工参与模式：

### 人工独占接管

- AI 停止对外发言
- 员工直接与微信用户沟通
- 节点可停止执行，或只保留后台分析能力

适用于：

- 投诉处理
- 敏感咨询
- 高风险业务沟通

### 人工审核协作

- AI 继续生成建议回复
- 员工审核后选择发送
- 对外发信权仍在网关，员工是最终确认者

适用于：

- 半自动客服
- 高价值客户
- 复杂业务咨询

## 6. 网关压力分析

## 6.1 网关承担的是“协调压力”，不是主要“推理压力”

网关主要负载包括：

- HTTP / WebSocket 连接管理
- Redis 会话状态读写
- 节点注册和心跳处理
- 控制台实时推送
- 员工接管状态机
- 微信消息入口与出站

这些大部分是：

- I/O 密集
- 短时逻辑判断
- 状态同步

真正重的部分通常仍在节点：

- 模型推理
- Dify / OpenAI 请求
- 工具调用
- 长文本处理

因此，网关会忙，但只要边界清晰，不应成为最重的算力组件。

## 6.2 网关容易被拖垮的几种错误设计

以下设计会让网关压力异常放大：

- 在网关直接做大模型推理
- 每次请求都读取完整历史 transcript
- 前端和节点都用高频轮询轰炸网关
- 消息、状态、诊断全部走同步串行流程
- 没有摘要层，所有请求都读取最重数据

## 7. 建议的优化方向

## 7.1 优先做长连接，而不是 P2P

推荐演进方向：

- 控制台 <- 网关：WebSocket 或 SSE
- 节点 -> 网关：WebSocket 或 gRPC stream

目标：

- 降低轮询开销
- 缩短状态传播时延
- 让会话事件、节点心跳、任务下发共享统一连接

## 7.2 做状态分层

推荐把数据分为三层：

### 实时态

- 当前会话消息流
- 当前节点诊断时间线
- 当前人工接管状态
- 当前任务执行状态

这些必须实时拉取或订阅。

### 摘要态

- 微信连接摘要
- 节点 summary
- 最近会话列表
- 当前在线节点数

这些可缓存、可后台刷新。

### 审计态

- 全量 transcript
- 配对日志
- 鉴权日志
- 人工接管历史

这些不参与高频实时刷新，主要用于追溯和排障。

## 7.3 让网关保持“薄协调层”

推荐原则：

- 网关只保留最小必要的运行状态
- 模型执行继续下放到节点
- 大量异步事件用 Redis / 队列承接
- 前端优先读取摘要，不要频繁拉全量明细

## 7.4 加强诊断与审计，而不是增加拓扑复杂度

每次关键操作建议带上统一 `trace_id`，覆盖：

- 微信消息接入
- 会话路由
- 节点执行
- 人工接管
- 微信出站

这样比引入 P2P 更能解决实际排障问题。

## 8. 推荐演进路线

### 阶段 1：稳住当前中心化模型

- 固定 `Gateway / Node / Console` 三端职责
- 明确员工接管状态机
- 统一所有出站消息必须经过网关

### 阶段 2：减少无效压力

- 前端摘要缓存 + 实时态分层
- 节点与控制台减少高频轮询
- 启动链路最小化

### 阶段 3：升级长连接

- 控制台改为 WebSocket / SSE 事件流
- 节点改为长连接注册、心跳、任务下发

### 阶段 4：补强异步化与容量治理

- 引入异步任务通道
- 做节点级限流与熔断
- 做网关摘要层和审计层拆分

### 阶段 5：再评估是否需要更复杂拓扑

只有在以下条件成立时，才考虑更复杂的网关集群或 P2P：

- 出现多网关协同需求
- 节点之间需要直接交换大流量数据
- 单网关经过长连接和异步化优化后仍然成为明确瓶颈

## 9. 当前推荐结论

对 `wechat-claw-hub` 当前阶段，推荐固定如下架构原则：

- 单网关中心化
- 节点主动回连网关
- 控制台只通过网关操作会话
- 员工参与会话时，网关仍是唯一裁决者与唯一出站通道
- 不优先做 P2P
- 优先做长连接、状态分层、异步化和诊断增强

这条路线更符合当前产品阶段，也更利于把“微信接入 + AI 执行 + 员工接管”三类复杂性收敛到一套可解释、可审计、可排障的运行模型中。
