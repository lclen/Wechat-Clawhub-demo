# 员工接管功能测试计划

## 1. 功能概述

员工接管（Employee Takeover）是指人工客服接管 AI 正在处理的会话，实现人机协作的关键功能。

### 1.1 核心能力

- **会话状态管理**：支持 `bot_active`、`handoff_pending`、`human_active`、`closing` 四种状态
- **节点切换**：支持手动切换会话到不同节点
- **消息角色区分**：支持 `user`、`bot`、`human`、`system` 四种消息角色
- **接管标识**：通过 `claimed_by` 字段记录接管人员

### 1.2 当前实现状态

**已实现**：
- ✅ 会话状态模型（`SessionStatus`）
- ✅ 消息角色模型（`MessageRole`）
- ✅ 会话切换 API（`POST /api/sessions/{session_id}/switch-node`）
- ✅ 前端接管状态展示（"人工中"、"待接管"标签）
- ✅ 前端切换节点按钮

**待验证**：
- ❓ 会话状态转换逻辑是否完整
- ❓ AI 与员工消息是否会冲突
- ❓ 接管后的会话释放机制
- ❓ 审计链是否完整

## 2. 测试场景

### 2.1 会话状态转换测试

#### 场景 1：正常接管流程

**前置条件**：
- 网关和节点正常运行
- 存在一个 `bot_active` 状态的会话
- 用户发送消息，AI 正在处理

**测试步骤**：
1. 用户在微信发送消息："我要转人工"
2. 系统识别转人工意图，将会话状态改为 `handoff_pending`
3. 员工在控制台看到"待接管"标签
4. 员工点击接管按钮（或发送消息）
5. 会话状态变为 `human_active`
6. `claimed_by` 字段记录员工 ID

**预期结果**：
- ✅ 会话状态正确转换：`bot_active` → `handoff_pending` → `human_active`
- ✅ 前端实时显示状态变化
- ✅ `claimed_by` 字段正确记录
- ✅ AI 停止处理该会话的新消息

**验证点**：
- 检查 Redis 中的会话记录
- 检查前端 WebSocket 推送的状态更新
- 检查节点是否停止接收该会话的任务

#### 场景 2：员工主动接管

**前置条件**：
- 存在一个 `bot_active` 状态的会话
- 员工在控制台查看会话列表

**测试步骤**：
1. 员工在控制台选择一个会话
2. 员工点击"切换节点"按钮
3. 系统调用 `POST /api/sessions/{session_id}/switch-node`
4. 会话状态变为 `human_active`

**预期结果**：
- ✅ 会话状态正确转换：`bot_active` → `human_active`
- ✅ 前端显示"人工中"标签
- ✅ AI 停止处理该会话

**验证点**：
- 检查 `switch_session_target` 方法的实现
- 检查是否有取消正在进行的 AI 任务的逻辑

#### 场景 3：会话释放

**前置条件**：
- 存在一个 `human_active` 状态的会话
- 员工已完成服务

**测试步骤**：
1. 员工点击"释放会话"按钮（或发送特定命令）
2. 会话状态变为 `bot_active`
3. `claimed_by` 字段清空
4. AI 恢复处理该会话

**预期结果**：
- ✅ 会话状态正确转换：`human_active` → `bot_active`
- ✅ `claimed_by` 字段清空
- ✅ AI 恢复接收该会话的任务

**验证点**：
- 检查是否有释放会话的 API
- 检查释放后 AI 是否能正常接收任务

### 2.2 消息互斥性测试

#### 场景 4：接管期间 AI 不发送消息

**前置条件**：
- 会话状态为 `human_active`
- 用户发送新消息

**测试步骤**：
1. 用户在微信发送消息："你好"
2. 系统接收消息，检查会话状态
3. 发现会话状态为 `human_active`
4. 系统不分发任务给 AI 节点

**预期结果**：
- ✅ AI 节点不接收该消息的处理任务
- ✅ 消息仅显示在员工的控制台
- ✅ 员工可以看到用户的新消息

**验证点**：
- 检查 `DispatchQueue` 的分发逻辑
- 检查是否有状态检查的代码
- 监控节点是否收到任务

#### 场景 5：员工发送消息后 AI 不抢答

**前置条件**：
- 会话状态为 `human_active`
- 员工正在回复用户

**测试步骤**：
1. 员工在控制台发送消息："您好，我是人工客服"
2. 消息角色标记为 `human`
3. 用户收到消息
4. 用户继续发送消息
5. AI 不处理用户的新消息

**预期结果**：
- ✅ 员工消息正确发送到微信
- ✅ 消息角色正确标记为 `human`
- ✅ AI 不抢答用户的后续消息

**验证点**：
- 检查消息发送 API 的角色参数
- 检查会话状态是否保持 `human_active`

### 2.3 并发场景测试

#### 场景 6：AI 处理中被接管

**前置条件**：
- 会话状态为 `bot_active`
- AI 正在处理用户消息（任务状态为 `inflight`）

**测试步骤**：
1. 用户发送消息："帮我查询订单"
2. AI 节点接收任务，开始处理
3. 员工在控制台点击"切换节点"
4. 会话状态变为 `human_active`
5. AI 节点完成处理，尝试发送回复

**预期结果**：
- ✅ 会话状态正确变为 `human_active`
- ✅ AI 的回复被拦截或标记为无效
- ✅ 用户不会收到 AI 的回复
- ✅ 员工可以看到 AI 的处理结果（可选）

**验证点**：
- 检查任务结果提交时的状态检查
- 检查是否有任务取消机制
- 检查消息发送前的状态验证

#### 场景 7：多员工同时接管

**前置条件**：
- 会话状态为 `handoff_pending`
- 两个员工同时查看该会话

**测试步骤**：
1. 员工 A 点击接管按钮
2. 员工 B 同时点击接管按钮
3. 系统处理并发接管请求

**预期结果**：
- ✅ 只有一个员工成功接管
- ✅ `claimed_by` 字段只记录一个员工 ID
- ✅ 另一个员工收到"已被其他员工接管"的提示

**验证点**：
- 检查是否有并发控制机制（如乐观锁）
- 检查 `claimed_by` 字段的更新逻辑

### 2.4 审计链测试

#### 场景 8：完整的接管审计链

**预期结果**：

**验证点**：
- 检查日志中的 `trace_id` 字段
- 检查会话状态变更的日志

## 4. 已知问题和改进建议

### 4.1 当前发现的问题

1. **缺少会话释放 API**
   - 当前只有 `switch-node` API，没有明确的"释放会话"API
   - 建议：添加 `POST /api/sessions/{session_id}/release` API

2. **缺少任务取消机制**
   - 接管时可能无法取消正在进行的 AI 任务
   - 建议：实现任务取消事件（`task_cancel_event`）

3. **缺少接管历史记录**
   - 无法查询会话的接管历史
   - 建议：添加接管历史查询 API

4. **缺少员工身份验证**
   - `claimed_by` 字段可能被任意设置
   - 建议：添加员工身份验证和权限控制

### 4.2 改进建议

1. **实现完整的状态机**
   - 定义所有合法的状态转换
   - 添加状态转换的验证逻辑
   - 记录状态转换的原因和操作人

2. **增强审计能力**
   - 扩展 `trace_id` 到所有关键操作
   - 添加结构化的审计日志
   - 实现审计日志查询 API

3. **优化前端体验**
   - 添加接管确认对话框
   - 显示当前接管人信息
   - 添加会话释放按钮
   - 实时显示员工在线状态

4. **添加监控指标**
   - 接管次数统计
   - 接管响应时间
   - 员工处理时长
   - AI 与员工切换频率
