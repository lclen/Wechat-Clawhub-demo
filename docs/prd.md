# wechat-claw-hub PRD

## 1. 项目目标

`wechat-claw-hub` 是一个用于承接微信用户消息、调度多个 `Claw` 节点、统一接入 Dify 知识库并支持人工接管的独立网关项目。

项目首版需要实现以下目标：

- 用户通过微信扫码接入系统
- 多个微信用户可同时访问同一个 agent
- 每个用户拥有独立上下文
- 消息由网关分发到多个 `Claw` 节点
- 各节点统一调用 Dify 知识库回答问题
- 用户要求转人工后，进入人工独占接管
- 人工在网页坐席台查看上下文并回复用户

## 2. 核心角色

### 微信用户

- 通过微信私聊与系统交互
- 可向同一个 agent 连续提问
- 在需要时可以申请转人工

### 网关服务

- 作为统一消息入口和统一出站通道
- 识别用户会话
- 维护上下文和人工接管状态
- 调度请求到合适的 `Claw` 节点

### Claw 节点

- 作为工作节点处理用户请求
- 每个节点有自己的并发上限
- 可同时处理多个用户请求
- 不负责保存用户长期上下文

### Dify 知识库

- 作为统一知识问答后端
- 由网关或节点统一调用 Dify API
- 为用户问题提供知识库答案

### 人工坐席

- 在网页坐席台查看用户会话
- 认领需要人工介入的会话
- 接管后继续与用户沟通
- 完成后释放会话给 AI

### 管理员

- 配置节点
- 查看系统状态
- 维护 Dify、Redis 和接入参数

## 3. 核心流程

### 微信扫码登录与消息接入流程

1. 管理员完成微信扫码接入配置
2. 微信用户向系统发送私聊消息
3. 微信消息被统一转换为内部消息格式
4. 网关接收并处理消息

### 用户消息进入网关后的会话识别流程

1. 网关从消息中提取微信私聊用户 ID
2. 使用用户 ID 作为主会话隔离键
3. 为新用户创建会话，为老用户恢复既有上下文
4. 将本轮消息写入 transcript 和 Redis 会话状态

### 节点负载均衡与分发流程

1. 网关读取所有 `Claw` 节点的状态、并发上限和当前占用
2. 优先考虑已有会话亲和的节点
3. 如果原节点繁忙或异常，则选择更空闲的健康节点
4. 将请求分发给目标节点处理

### 会话级切换与槽位复用流程

1. 每个 `Claw` 节点上报 `channel_capacity`
2. 网关为会话分配一个活动 `slot_id`
3. 同一微信用户在任一时刻最多绑定一个活动槽位
4. 用户在微信输入 `/switch` 或“切换节点”时，网关释放当前槽位并重新分配
5. 切换过程中主会话、上下文摘要、长期记忆和 transcript 不重置
6. 用户 10 分钟无回复时，只释放当前槽位，不关闭主会话

### Dify 问答流程

1. 节点接收到网关分发的请求
2. 节点从网关获取该用户独立上下文
3. 节点统一调用 Dify API
4. 得到答案后返回网关
5. 网关再将消息发回微信用户

### 人工接管申请、认领、回复、释放流程

1. 用户表达转人工意图，或 agent 判断需要人工处理
2. 系统触发人工接管工具
3. 网关将会话状态切换为待人工处理
4. 网页坐席台展示待接管会话
5. 人工认领后，会话进入人工独占接管状态
6. AI 自动回复暂停
7. 人工查看聊天记录并继续回复用户
8. 人工完成后释放会话，AI 恢复处理

### 节点故障或超载时的转移流程

1. 网关持续检查节点健康状态
2. 当节点异常、超时或达到并发上限时，停止为其分配新请求
3. 新请求被分发到其他健康节点
4. 由于上下文由网关统一持有，用户会话可以继续进行

## 4. 产品规则

- 同一个 agent 服务多个用户
- 上下文按微信私聊用户 ID 隔离
- 一个用户在任一时刻只能处于一个主会话
- 网关是唯一会话状态持有者
- `Claw` 节点不持有用户长期上下文
- 同一个用户的多轮消息必须命中同一个会话上下文
- 同一个用户的执行资源以 `node_id + slot_id` 表示，可随时切换
- 人工接管后 AI 自动回复暂停
- 人工释放后 AI 恢复工作
- 人工消息与 AI 消息都必须经过网关统一发出
- 开启分发模式后，宿主机本机节点不再参与处理，只保留网关接入与调度能力

## 5. 首版页面需求

### 登录/系统状态页

- 展示系统是否启动
- 展示 Redis、Dify、微信接入和节点状态

### 会话列表页

- 查看当前会话列表
- 区分 AI 会话、待接管会话、人工接管中会话

### 会话详情聊天页

- 查看用户聊天记录
- 查看系统事件
- 查看当前会话归属节点和接管状态

### 人工认领与释放操作

- 人工可认领待接管会话
- 人工可释放已接管会话

### 用户聊天记录查看

- 展示用户发送的历史消息
- 展示 AI 回复消息
- 展示人工回复消息
- 展示转人工、节点切换等系统事件

### 人工发送消息输入框

- 人工可直接输入并发送消息给用户
- 消息由网关中转发往微信

### 节点状态面板

- 查看每个 `Claw` 节点的健康状态
- 查看并发上限、当前占用和最近错误
- 查看通道容量、通道占用
- 查看节点 ID、节点名称、上报地址、平台、版本
- 若节点上报真实局域网地址，则在接入中心直接显示局域网 IP

### 基础告警与错误提示

- 显示节点异常
- 显示 Dify 调用异常
- 显示微信发送失败

### 当前前端结构约束

- 接入中心用于查看模型、微信、节点接入状态
- 会话观察台采用“左侧会话列表 + 中间聊天记录 + 右侧会话记忆抽屉”的工作台结构
- 会话记忆默认收起，通过“当前会话”区域右侧按钮打开
- 会话列表需优先展示用户显示名和平台用户 ID，而不是把原始 session_id 作为主标题
- 会话观察台需展示当前节点、当前槽位、路由模式和“切换节点”入口
- 接入中心需提供“分发模式”开关，并在无可用远端节点时明确提示风险

## 6. 技术实现大纲

### 网关层

- 统一消息入口
- 统一消息出站
- 会话管理
- Redis 上下文读写
- 节点负载均衡调度
- 会话级槽位租约管理
- `/switch` 命令解析与切换
- 人工接管状态机
- transcript 与事件日志

### 节点层

- 作为无长期状态的工作节点
- 每个节点受并发限制
- 每个节点具备有限数量的可复用通道槽位
- 节点从网关获取当前请求所需上下文
- 节点统一调用 Dify 或 OpenAI 兼容模型

### 存储层

- Redis 保存上下文
- Redis 保存会话状态
- Redis 保存节点占用
- Redis 保存人工接管状态
- transcript 记录消息和系统事件

### 通信层

- 微信接入到网关
- 网关到节点通过 HTTP 或内部 RPC
- 网页坐席台通过 HTTP + WebSocket 与网关通信
- 微信个人号链路当前已具备长轮询、上下文 token 管理、typing 提示能力
- 节点发现与配对阶段通过 UDP 广播 + pairing key 建立受控连接

### 审计层

- 保存消息 transcript
- 保存人工接管事件
- 保存节点分发日志
- 保存错误日志和调用链路信息

## 7. 非功能要求

- 支持多用户并发
- 支持节点健康检查与故障切换
- 支持消息可追踪
- 支持人工接管全链路可审计
- 支持服务重启后的会话恢复
- 支持在同一个 agent 下服务多个用户且上下文不串线
- 支持即使同一用户被重新分发到不同节点，也保持会话记忆连续

## 8.1 当前微信字段能力边界

当前首版基于 iLink 消息事件接入，稳定可获取的信息包括：

- `from_user_id`
- `message_id`
- `session_id`
- `context_token`
- 文本与媒体消息内容

当前未稳定获取的信息包括：

- 微信号
- 微信昵称
- 备注名
- 头像
- 联系人详情

因此前端当前展示的用户身份信息，本质上是平台返回的 `from_user_id`，不是用户设置的微信号。

## 8. 验收标准

- 两个以上用户同时对话时上下文不串
- 多个用户可以同时接入同一个 agent
- 单节点忙碌时新消息能分发到其他节点
- Dify 可稳定返回知识库答案
- 用户提出转人工后，网页坐席台能继续对话
- 人工释放后 AI 可恢复接待
- 服务重启后会话状态与上下文可恢复
- 接入中心可看到已接入节点及其节点 ID、上报地址、平台、版本和状态
- 会话观察台可打开右侧会话记忆抽屉查看自动记录的会话记忆
- 微信用户在同一会话输入 `/switch` 后，可切到新的处理节点或新的可用槽位，且上下文连续

## 8.2 当前实现状态补充

- 已实现：
  - 微信扫码接入、长轮询、typing 提示
  - 节点注册、心跳、拉任务、结果回传
  - 节点 `channel_capacity / channel_in_use` 上报
  - 会话 `assigned_node_id / assigned_slot_id / slot_expires_at` 状态
  - 控制台手动“切换节点”
  - 微信端 `/switch` / “切换节点”命令切换
  - 分发模式开关
- 当前仍需继续打磨：
  - `/switch` 更偏“重分配到新的可用节点/槽位”，不是严格随机
  - 人工接管完整闭环仍未完成
  - 真实联系人资料能力仍未接入
  - 模型检测当前检测的是网关内置模型，而不是节点模型

## Architecture Defaults

以下默认决策直接写入，不再留给实现阶段二次判断：

- 项目名：`wechat-claw-hub`
- 项目位置：`D:\wechat-claw-hub`
- 后端栈：`Python 3.11+ + FastAPI`
- 上下文存储：`Redis + 持久化`
- 会话隔离粒度：按微信私聊用户 ID
- 节点模型：有并发上限的工作节点，由网关统一调度
- 调度策略：会话亲和 + 最小负载优先
- 知识库接法：统一 Dify API
- 人工模式：人工独占接管
- 人工端首版：网页坐席台
- 企业微信：可后续补做提醒入口，但不是首版主入口
